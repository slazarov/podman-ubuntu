CAUSE 1 — CODE BUG in functions.sh detect_runtime_depends() (currently line ~145, the loop is fed by `ldd "${bin}" | awk '/=> \// {print $3}'`). `ldd` resolves the FULL transitive shared-object closure (deps-of-deps), but Debian policy (dpkg-shlibdeps) derives Depends from a binary's DIRECT DT_NEEDED entries only. Change the loop so it resolves ONLY the binary's direct DT_NEEDED sonames to paths, then keeps the EXISTING resolution chain unchanged:

  1. Enumerate the binary's direct DT_NEEDED sonames. Use `objdump -p "${bin}"` and extract the soname field from `NEEDED` lines (e.g. `objdump -p "${bin}" 2>/dev/null | awk '/NEEDED/{print $2}'`). `readelf -d "${bin}"` is an acceptable equivalent if objdump is unavailable, but prefer objdump for parse stability. The result is a set of bare sonames like `libgpgme.so.11`, `libseccomp.so.2` — NOT full paths.
  2. For EACH direct soname, resolve it to the absolute on-disk path of the library THIS binary would load. Resolve via the dynamic-loader search: parse `ldd "${bin}"` output and match the soname to its `=> /path` resolution (the soname is field 1 of the `soname => /path` ldd line), so the resolved path is the exact object the binary links. Equivalent: `ldconfig -p` lookup of the soname → realpath. Prefer the per-binary ldd match because it reflects what this binary actually resolves (multiarch correctness). Skip the linux-vdso and ld-linux loader entries (no resolved path), exactly as today.
  3. Feed each resolved path through the EXISTING, UNCHANGED chain: `realpath` normalize (Pitfall 4) → `dpkg-query -S` run OUTSIDE a pipeline with its exit status tested (CR-01, WR-01 — preserve the diversion-line filtering + `head -n1` + `awk -F:` package-field parse) → accumulate into the `pkgs` associative array → final libc6/libgcc-s1 exclusion (D-02) → `sort -u`.

  - D-03: any direct NEEDED soname that resolves to a path with NO owning package still hard-fails (return 1) with the existing ERROR message. No `|| true`, no silent skip. If a NEEDED soname cannot be resolved to a path at all (not found by ldd/ldconfig), that is also a hard-fail with a clear ERROR naming the soname and the linking binary — a NEEDED soname that does not resolve is a real breakage, not something to swallow.
  - D-01/D-04: do NOT hardcode any soname→package mapping. The package name still comes only from `dpkg-query -S` against the host dpkg DB. The crun JSON-parser dep must still fall out automatically (no parser special case).
