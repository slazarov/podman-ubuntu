<objective>
Retrofit the `publish` job in `.github/workflows/build-packages.yml` so it (a) runs only when ALL FOUR matrixed build cells succeed (atomic publish — CICD-08), (b) downloads each distro's artifacts into a distro-scoped directory with no cross-distro merge (CICD-07/CICD-05), and (c) invokes `ci_publish.sh` once per distro with that distro's compact label and only that distro's `.deb` files. Add a pure-bash/YAML unit test (`tests/test_ci_matrix.sh`) that asserts the matrix shape, `fail-fast: false`, the publish-gating condition, and the no-merge download contract so the whole Phase-21 wiring is provable on the macOS dev host without running CI.

Purpose: Plan 21-01 built the four-cell matrix and per-distro artifacts but deliberately left the publish job pointing only loosely at `build`. The publish path from Phase 20 (`ci_publish.sh <track> <distro> <deb-dir> <repo-url> <out>`) is already per-distro; this plan drives it correctly for both distros from one publish job while preserving the single atomic GitHub Pages deploy (one `upload-pages-artifact` + `deploy-pages` at the end, so the live repo is replaced in one shot or not at all).

Output: A publish job that gates on the matrixed `build` job's overall success, downloads `debs-2404-*` and `debs-2604-*` into separate directories, runs `ci_publish.sh` for 2404 then 2604 into a single accumulating `repo-output`, and deploys once. Plus `tests/test_ci_matrix.sh` proving the contract.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build cells → publish job | Per-distro artifacts cross into the publish job; a wrong-distro merge would mis-route binaries into the wrong suite |
| matrix aggregate result → publish gate | The build job's `result` decides whether the live repo is touched at all |
| downloaded artifacts → ci_publish.sh deb-dir | The deb directory passed to ci_publish.sh determines which binaries enter which suite |
| repo-output → GitHub Pages (deploy-pages) | The single deploy replaces the live repository in one shot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-05 | Tampering | Cross-distro artifact merge in publish | mitigate | Two separate download steps (`debs-2404-*`→`all-debs-2404/`, `debs-2604-*`→`all-debs-2604/`); no step uses bare `pattern: debs-*`; each distro's ci_publish.sh run is fed only its own directory. Test assertion 9 enforces no bare `debs-*` pattern. |
| T-21-06 | Denial of Service / Elevation | Publish-gate bypass on partial failure | mitigate | `if: always() && needs.build.result == 'success'` — a single matrix-cell failure makes the aggregate `build.result` non-success, so publish is skipped and the live repo is untouched (CICD-08). Test assertion 8 enforces the gating expression. |
| T-21-07 | Injection | distro label → ci_publish.sh/config.sh | mitigate | The publish job passes the fixed compact labels `2404`/`2604` only (string literals in the workflow, not free-form input); `resolve_publish_targets`/`is_valid_suite` re-validate against `VALID_DISTROS=(2404 2604)` and abort on anything else |
| T-21-08 | Tampering | second ci_publish.sh run clobbering first distro's suites | mitigate | ci_publish.sh's mirror-then-include model (`OTHER_SUITES = ALL_SUITES − PUBLISH_TARGETS`, Phase-20 verified 62/62) means the 2604 run mirrors the 2404 run's freshly-published suites as OTHER_SUITES into the same repo-output rather than overwriting them; 2404-first ordering is explicit |
| T-21-SC | Tampering | apt install of reprepro in publish | accept | Unchanged from Phase 20 — `reprepro` from Ubuntu's signed archive; no new package source, no language-package-manager installs, so no `## Package Legitimacy Audit` is required |
</threat_model>
