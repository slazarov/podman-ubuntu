<objective>
Collapse the two duplicated build jobs (`build-amd64`, `build-arm64`) in `.github/workflows/build-packages.yml` into a single matrixed `build` job that fans out to all four distro×arch cells (24.04/26.04 × amd64/arm64). The 26.04 cells run inside `ubuntu:26.04` containers on the existing native amd64/arm64 runners; cache keys and artifact names gain a distro dimension so no 26.04 binary can leak into a 24.04 package or vice versa.

Purpose: Phase 20 already threaded a `distro` dimension through the publish path (`ci_publish.sh`/`repo_manage.sh` take `(track, distro)`; `config.sh` validates `2404`/`2604`; `detect_distro_version_id` honors the dotted `DISTRO` override). The build half of the pipeline still only produces 24.04 packages from two hardcoded-arch jobs. This plan converts the build half to a four-cell matrix so native 26.04 packages get built (CICD-05/06/07).

Output: A rewritten `build-packages.yml` with one `build` matrix job (4 cells, `fail-fast: false`), distro-dimensioned Go caches (`go-<distro>-<arch>-<track>-<run_number>`) and artifacts (`debs-<distro>-<arch>`), and runner-agnostic container selection for the 26.04 cells. The `publish` job is retrofitted in Plan 21-02 — this plan leaves `publish` working against the new artifact names only to the extent required to not break YAML parsing (its `needs`/gating rewrite is Plan 02's job).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| upstream git → CI build cells | Untrusted upstream source (nightly = HEAD) compiled inside each cell |
| matrix value → shell env (DISTRO) | Matrix-supplied distro label expanded into env passed to setup.sh/package_all.sh |
| build cell → Go cache (actions/cache) | Cache restored into a build cell can poison the produced binary |
| build cell → artifact (upload-artifact) | Artifact name determines which suite a binary later lands in (publish, Plan 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01 | Tampering | Go cache cross-distro reuse | mitigate | Cache key includes `matrix.distro` AND `matrix.arch` (`go-<distro>-<arch>-<track>-<run>`), so a 26.04 cell can never restore a 24.04 cell's cache; restore-keys are scoped to the same distro+arch prefix only |
| T-21-02 | Spoofing/Tampering | Artifact cross-contamination | mitigate | Each cell uploads a distinctly named artifact `debs-<distro>-<arch>`; no cell writes another cell's artifact name. (The download-side no-merge guarantee is Plan 02's T-21-05.) |
| T-21-03 | Injection | matrix.distro → DISTRO env | mitigate | Matrix values are a fixed in-workflow `include` allowlist (`2404`/`2604` only — not free-form input); the dotted form is derived via a closed case map (`2604`→`26.04`); `detect_distro_version_id` re-validates against `^[0-9]+\.[0-9]+$` (T-19-01 fail-closed) before it reaches any filename |
| T-21-04 | Tampering | 26.04 container base image | accept | `ubuntu:26.04` is the official Docker Hub Ubuntu image pulled by GitHub's container runtime; pinning by digest is deferred (the same trust model as the host runner images already in use). No new third-party action is introduced. |
| T-21-SC | Tampering | apt installs inside ubuntu:26.04 container bootstrap | mitigate | The container bootstrap installs only distro-archive packages (`sudo git curl ca-certificates`) from Ubuntu's signed archive via `apt-get` — no npm/pip/cargo, no third-party package source. No `## Package Legitimacy Audit` is required because no language-package-manager installs are added. |
</threat_model>
