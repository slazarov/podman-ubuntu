<objective>
Build the post-export Acquire-By-Hash bolt-on (REPO-08 / D-06/D-07/D-08): a self-contained helper (`scripts/repo_byhash.sh`, exposing `add_byhash_and_resign`) that, per suite after `reprepro export`, (a) materializes `by-hash/<ALGO>/<hash>` copies of every index file adjacent to that index plus a by-hash copy of `Release`, (b) injects `Acquire-By-Hash: yes` into the suite's `Release`, and (c) re-signs `InRelease` + `Release.gpg` with the already-imported GPG key. Ship the `test_byhash_parse.sh` unit test that pins the Release-section parser against a literal fixture.

Purpose: reprepro has no native Acquire-By-Hash support (VERIFIED Debian bug #820660), and GitHub Pages' CDN can serve a stale index alongside a fresh Release mid-deploy, producing apt hash-sum mismatches. By-hash + re-sign eliminates that race. Re-signing is mandatory (D-08): editing Release after export invalidates reprepro's existing InRelease/Release.gpg, so without it apt rejects the repo with BADSIG. This helper is the only thing reprepro cannot do; it is the smallest possible `cp` + `gpg` wrapper around reprepro's own Release output. Plan 03 calls it per-suite.

Output: `scripts/repo_byhash.sh` providing `add_byhash_and_resign`, and a passing `tests/test_byhash_parse.sh`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reprepro Release output -> by-hash copies | by-hash files are byte-identical copies of canonical indexes whose hashes are signed in Release |
| mutated Release -> apt-secure signature chain | Editing Release after export invalidates reprepro's signatures; re-sign restores the trust anchor |
| imported GPG key in keyring -> InRelease/Release.gpg | the single private key (offline, never on Pages) signs the served metadata |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-04 | Tampering/Spoofing | Stale signature after Acquire-By-Hash injection | mitigate | Mandatory re-sign step (D-08, Pitfall 2): `rm -f InRelease Release.gpg` then regenerate both with the imported key. Task 1 acceptance enforces both gpg commands and that they run after injection. Without it apt reports BADSIG. |
| T-20-05 | Tampering | Drift between by-hash copy and served Release | mitigate | The Release by-hash copy is computed AFTER injection (Pitfall 2 ordering), so the hashed bytes equal the served bytes; index by-hash copies are `cp -f` of the canonical file whose hash reprepro already signed (no new trust surface). |
| T-20-06 | DoS (failed updates) | CDN hash-sum mismatch race | mitigate | Acquire-By-Hash adjacent to each index (Pitfall 3) lets apt fetch the exact signed index version even if the CDN serves a stale Release momentarily; this is the requirement's purpose. |
