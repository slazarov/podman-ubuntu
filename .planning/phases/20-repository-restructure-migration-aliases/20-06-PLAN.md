<objective>
Close verification GAP 2 (CR-02, BLOCKER): stop `scripts/ci_publish.sh` from
re-signing the bare legacy aliases (stable/edge/nightly) on a 26.04 publish. Today,
for `DISTRO=2604` the bare alias is not a publish target, so it lands in
`OTHER_SUITES`, gets re-`includedeb`'d from mirrored debs and re-`export`ed with a
fresh reprepro db — regenerating its Release Date + signature despite byte-identical
package content. That reopens the CDN hash-mismatch window Acquire-By-Hash exists to
prevent. Fix it by serving the unchanged alias's `dists/` tree verbatim (original
signatures preserved) instead of re-exporting it. Fold in the cheap WR-04 HTML
escaping while in this file.

Purpose: REPO-08 (SC-3) requires no CDN hash-sum mismatches on any suite, and SC-4
requires routing without clobbering other suites — both currently fail for the bare
aliases on 26.04 publishes. This plan makes the 26.04 publish path preserve signature
stability for suites it does not change, and adds the missing 26.04 test coverage.

Output: a verbatim-mirror path for non-target bare aliases in `ci_publish.sh`;
HTML-escaped package names/versions in the generated index.html; a new Test group G
in `tests/test_repo_assemble_byhash.sh` that publishes a 26.04 track and asserts the
bare alias Release Date/signature are preserved.
</objective>

<design_constraints>
Honor the locked CONTEXT decisions:
- D-10/D-13: keep the rebuild-the-world mirror-then-include model. Do NOT introduce
  a persistent reprepro db or an incremental gh-pages checkout. The fix is to mirror
  the UNCHANGED alias's already-signed `dists/` tree verbatim rather than re-exporting
  it — this stays within the rebuild-the-world model, it just stops mutating suites
  the publish did not change.
- D-12: a 24.04 publish must still feed BOTH `<track>-2404` and the bare `<track>`
  alias from fresh debs. The verbatim-mirror path applies ONLY to suites that are
  not publish targets (i.e. the bare aliases on a 26.04 publish, and any other
  untouched versioned suite for which we already mirror down debs).
- D-14: empty-but-signed `-2604` suites and first-deploy tolerance (a 404 on the
  live Packages/Release URL) must keep working — verbatim-mirror must no-op cleanly
  when the live alias tree does not exist yet.
The verifier offered two options (treat the alias as a first-class verbatim-mirror
target, OR guarantee Release Date/signature stability some other way). Choose the
verbatim-mirror approach — it is the lower-risk fix and matches D-10's model.
</design_constraints>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| live GitHub Pages repo → CI publish assembly | untouched suites' already-signed metadata is read here; preserving it verbatim is what keeps apt clients' cached signatures valid |
| CI publish assembly → GitHub Pages CDN (apt clients) | Release/InRelease/Release.gpg + by-hash indexes cross here; a regenerated signature on an unchanged suite opens a hash-mismatch window |
| upstream package version strings → generated index.html | nightly builds derive versions from upstream HEAD; unescaped values cross into served HTML |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-17 | Tampering | bare-alias re-export on 26.04 publish (ci_publish.sh) | mitigate | Serve the unchanged alias's dists tree verbatim instead of re-includedeb + re-export + re-sign; the served Release Date/signature stay byte-identical so a CDN mid-deploy cannot pair a new InRelease with a stale by-hash index. Test group G proves byte-stability. |
| T-20-18 | Information Disclosure / Tampering | index.html package/version interpolation (ci_publish.sh:388-404) | mitigate | HTML-escape `& < >` in package names and versions before interpolation; closes the XSS / malformed-markup vector for nightly versions derived from attacker-influenceable upstream HEAD tags (WR-04). |
| T-20-19 | Denial of Service | first-deploy / empty-2604 publish path | accept | Verbatim-mirror no-ops cleanly when the live alias tree 404s (reuses the existing `curl ... || true` empty-check); the empty-but-signed D-14 behavior is unchanged. Low risk, well-covered by existing group E. |
| T-20-SC | Tampering | npm/pip/cargo installs | mitigate | No new package-manager installs — only Bash edits and a test on the distro-packaged `reprepro`. No legitimacy checkpoint required. |
</threat_model>
