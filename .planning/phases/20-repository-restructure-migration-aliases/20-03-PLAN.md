<objective>
Wire the foundation pieces (Plan 01 routing helper + 9-stanza distributions, Plan 02 by-hash helper) into the live publish path. Extend `scripts/repo_manage.sh` to accept the distro dimension and feed both `<track>-2404` and the bare alias on a 24.04 publish (D-11/D-12); generalize `scripts/ci_publish.sh` to mirror down the N untouched suites, reassemble all 9 suites without clobbering (D-10/D-13), call `add_byhash_and_resign` per suite (D-07), and iterate the 9-suite set in the index.html loop (D-18); and plumb the new `distro` argument through the publish job in `.github/workflows/build-packages.yml`.

Purpose: This is the integration spine that turns the 9-suite metadata + routing contract into an actual deployable multi-suite repo. The mirror-then-include pattern (every publish reassembles all 9 suites, only the published suite — and for 24.04 its alias — taking new content) is what satisfies Success Criterion 4 (no clobbering). Per-suite `reprepro export <suite>` discipline is preserved (Pitfall 4 — a bare `export` would wipe the target suite's Packages). The by-hash call lands after each export so the signed Release matches the on-disk index (REPO-08). The CI plumbs only the single `distro` argument; the matrix fan-out that supplies multiple distro values is Phase 21.

Output: A track+distro-aware `repo_manage.sh`, a 9-suite mirror-then-include `ci_publish.sh` with by-hash post-processing, and a publish job that passes `distro` to `ci_publish.sh`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| live repo URL -> mirror-down -> reassembled tree | untouched suites' .debs are fetched from the live CDN and re-included; reprepro re-verifies their hashes on includedeb |
| publish routing -> which suite gets fresh content | a wrong target could overwrite or omit a suite (clobbering) |
| CI workflow args -> ci_publish.sh | the distro value selects the target suite + whether the alias is fed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-07 | Tampering | Clobbering untouched suites | mitigate | Mirror-then-include reassembles all 9 suites; OTHER_SUITES = ALL_SUITES − PUBLISH_TARGETS, so only the published suite(s) take fresh content (D-13). Per-suite `export <suite>` (Pitfall 4) prevents a bare export wiping populated suites. Task 1/Task 2 acceptance enforce both. |
| T-20-08 | Tampering/Spoofing | by-hash applied before re-sign / skipped on some suites | mitigate | `add_byhash_and_resign` (Plan 02) is invoked for every suite that has a Release, after all exports — so no suite is left with an unsigned Acquire-By-Hash mutation. |
| T-20-09 | Elevation/Spoofing | malformed publish target from CI input | mitigate | `resolve_publish_targets` validates track+distro before any includedeb; an out-of-whitelist value aborts the publish rather than creating a rogue suite. |
| T-20-10 | DoS (failed first deploy) | new -2404/-2604 Packages URLs 404 on first 9-suite publish | accept | The existing `curl -sfL ... || true` + empty-Packages-continue tolerance (A3) treats a 404 as "first deploy / not yet published"; the suite is then created empty-but-signed by reprepro export. Preserved verbatim in Task 2. |
| T-20-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan (bash wiring + workflow YAML; reprepro/gpg/curl already in the proven pipeline). RESEARCH Package Legitimacy Audit: N/A. |
</threat_model>
