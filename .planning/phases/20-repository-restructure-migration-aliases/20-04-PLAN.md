<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| deployed Pages CDN -> real apt client | the production trust anchor: apt verifies InRelease signature + per-index hashes against the served metadata |
| pre-v3.0 client cached Suite -> served alias Suite | apt refuses a silent Suite change; the bare alias must serve `Suite: stable` |
| throwaway test key -> assembled fixture repo | the harness signs with an ephemeral key in an isolated GNUPGHOME, never the production key |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-20-11 | Spoofing | apt Suite-change re-acceptance prompt on legacy client | mitigate | The deployed bare alias serves `Suite: stable`; Task 2 step 3 asserts the absence of the "changed its 'Suite' value" line in the real `apt update` log — the definitive REPO-07 proof against a real pre-v3.0 client (D-15). |

| T-20-13 | Tampering | Test key contaminating the host keyring | mitigate | The harness uses an isolated `GNUPGHOME` under mktemp with EXIT-trap cleanup; the production `GPG_PRIVATE_KEY` is never used in the test. |
